# Renesas RA CMake Project

This project is generated by RASC.

###  Building via CLI:
Configure: ```cmake -DARM_TOOLCHAIN_PATH="/your/toolchain/path" -DCMAKE_TOOLCHAIN_FILE=cmake/gcc.cmake  -G Ninja -B build/Debug```

- Ex.: ```cmake -DARM_TOOLCHAIN_PATH="C:/12_2_mpacbti_rel1/bin" -DCMAKE_TOOLCHAIN_FILE=cmake/gcc.cmake  -G Ninja -B build/Debug```
- Ex.: ```cmake -DARM_TOOLCHAIN_PATH="C:/12_2_mpacbti_rel1/bin" -DCMAKE_TOOLCHAIN_FILE=cmake/gcc.cmake  -DCMAKE_BUILD_TYPE=Release -G Ninja -B build/Release```

Build: ```cmake --build build/Debug```

### IRremoteESP8266 RA4M2 port

This project links a small RA4M2 port layer for IRremoteESP8266 send and receive support.
Before configuring the project, clone IRremoteESP8266 into `third_party`:

```sh
cd Renesas_IRremote
mkdir -p third_party
git clone https://github.com/crankyoldgit/IRremoteESP8266.git third_party/IRremoteESP8266
```

The LGPL-2.1 license is provided by the cloned repository at
`third_party/IRremoteESP8266/LICENSE.txt`.

GlobalCache, Pronto, and Sherwood send support are also enabled. Upstream treats GlobalCache and Pronto as send-only formats, and Sherwood decoding as NEC-compatible rather than a separate decoder.

Receive also enables the upstream hash fallback for unknown protocols.

The RA4M2 compatibility layer is in `src/ir_ra4m2` and provides the Arduino API surface that these protocols need:

- `pinMode`
- `digitalWrite`
- `digitalRead`
- `micros`
- `millis`
- `delayMicroseconds`
- `delay`
- `yield`

Pins are FSP pin tokens, not Arduino board numbers. For example:

```cpp
IRsend irsend(BSP_IO_PORT_00_PIN_05);
irsend.begin();
irsend.sendNEC(irsend.encodeNEC(0x00FF, 0x10));
```

`hal_entry.cpp` contains disabled send and receive demos. Enable the NEC send demo at configure time:

```sh
cmake -DARM_TOOLCHAIN_PATH="/your/toolchain/path" \
      -DCMAKE_TOOLCHAIN_FILE=cmake/gcc.cmake \
      -DCMAKE_BUILD_TYPE=Debug \
      -DIRREMOTE_RA4M2_SEND_DEMO=ON \
      -DIRREMOTE_RA4M2_DEMO_TX_PIN=BSP_IO_PORT_00_PIN_05 \
      -G Ninja -B build/Debug
cmake --build build/Debug
```

Enable the receive demo with:

```sh
cmake -DARM_TOOLCHAIN_PATH="/your/toolchain/path" \
      -DCMAKE_TOOLCHAIN_FILE=cmake/gcc.cmake \
      -DCMAKE_BUILD_TYPE=Debug \
      -DIRREMOTE_RA4M2_RECV_DEMO=ON \
      -DIRREMOTE_RA4M2_DEMO_RX_PIN=BSP_IO_PORT_00_PIN_05 \
      -G Ninja -B build/Debug
cmake --build build/Debug
```

The receive demo updates these volatile globals when a frame is decoded:

- `g_irremote_last_decode_type`
- `g_irremote_last_value`
- `g_irremote_last_bits`

Current receive support uses a SysTick-based sampler by default because this empty FSP project does not allocate an ICU external interrupt vector or a timer driver instance. `IRrecv::enableIRIn()` configures SysTick at 50 us intervals, samples the active-low receiver pin, captures edge durations with `micros()`, and leaves `IRrecv::decode()` non-blocking until a frame has timed out and is ready to decode.

If SysTick is already reserved by another framework, disable the sampler at configure time:

```sh
cmake -DIRREMOTE_RA4M2_USE_SYSTICK_RECV=OFF ...
```

With `IRREMOTE_RA4M2_USE_SYSTICK_RECV=OFF`, `IRrecv::decode()` falls back to the older blocking polling capture path. For production systems that already allocate an FSP ICU external interrupt and hardware timer, replacing the SysTick sampler with that board-specific backend is still the better long-term integration.

All upstream `src/ir_*.cpp` protocol sources are compiled from
`third_party/IRremoteESP8266/src` by default. To use a checkout in another
location, configure with `-DIRREMOTEESP8266_DIR=/path/to/IRremoteESP8266`.
`_IR_ENABLE_DEFAULT_` remains false and the RA project explicitly enables the
send/decode protocols listed above; the only upstream decode macros left off are
formats upstream marks as not applicable (`GlobalCache`, `Pronto`, `Sherwood`)
plus the internal `DECODE_AC` aggregate macro, which upstream derives automatically
when A/C decoders are enabled.


### Configure via Visual Studio Code
- Set ARM_GCC_TOOLCHAIN_PATH as an environment variable before starting VS code or alternatively set ARM_TOOLCHAIN_PATH in .vscode/cmake-kits.json
- Select "ARM GCC kit with toolchainFile" kit in VS Code status bar
- It is recommended to avoid spaces in the toolchain and project paths as they might be interpreted as delimiters by CMake and the other build tools.

Example:

```set ARM_GCC_TOOLCHAIN_PATH=C:/12_2_mpacbti_rel1/bin```
```cd "/home/zheng/Github/ir_learning/workspace/Renesas_IRremote" && code .```

- Click build in VS Code status bar
